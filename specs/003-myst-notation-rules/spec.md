# Feature Specification: MyST記法ルールの拡充（Mermaid対応・追加ルール）

**Feature Branch**: `003-myst-notation-rules`
**Created**: 2026-02-16
**Status**: Draft
**Input**: User description: ".claude/skills/myst/SKILL.mdをもとにmyst記法の仕様を策定。Mermaidダイアグラム対応を追加し、追加ルールを提案する。"

## 背景と目的

本プロジェクトでは `docs/index.md` をはじめとするMyST形式のスライド・ドキュメントを作成する。これらのコンテンツを一貫した品質で記述するためには、MyST記法のルールが `.claude/skills/myst/SKILL.md` に定義されている必要がある。

本仕様は、SKILL.mdに定義すべきルールの内容を策定するものである。SKILL.mdはコンテンツ作成者（人間およびAI）がMyST記法でドキュメントを記述する際に参照するルールブックとして機能する。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - フローや関係性をダイアグラムで表現する (Priority: P1)

コンテンツ作成者（または生成AI）がワークフロー、シーケンス、状態遷移などの流れをドキュメントやスライドに記述する際、ASCIIアートやテキストベースの図ではなく、Mermaidディレクティブを使って構造化されたダイアグラムとして表現する。

**Why this priority**: フローチャートやシーケンス図はスライドの中核的なビジュアル要素であり、ASCIIアートからの置き換えによるビジュアル品質とメンテナンス性の改善インパクトが最も大きい。

**Independent Test**: 1つのフローチャートをMermaidに変換し、ビルドしてスライド上で正しくレンダリングされることを確認できる。

**Acceptance Scenarios**:

1. **Given** ASCIIアートで記述されたフローチャートがある, **When** ルールに従ってMermaidディレクティブに書き換える, **Then** ビルド後にダイアグラムとして視覚的にレンダリングされる
2. **Given** 新しいフロー図を追加する必要がある, **When** コンテンツ作成者がルールを参照する, **Then** `{mermaid}`ディレクティブの正しい構文で記述できる
3. **Given** Mermaidで表現できない複雑な図がある, **When** ルールを参照する, **Then** 代替手段（画像ファイルなど）を使うべきケースが明確に判断できる

---

### User Story 2 - テーブルとコードを構造化ディレクティブで記述する (Priority: P2)

コンテンツ作成者がテーブルデータを記述する際はlist-tableディレクティブを使用し、外部ファイルのコードを取り込む際はliteralincludeディレクティブを使用する。これにより一貫したフォーマットでコンテンツを生成する。

**Why this priority**: テーブルとコード取り込みはスライドやドキュメントの基本構成要素であり、記法の統一がドキュメント全体の品質に直結する。

**Independent Test**: パイプテーブルやインラインコードブロックが含まれるドキュメントに対してルールを適用し、list-table/literalincludeに変換できることを確認する。

**Acceptance Scenarios**:

1. **Given** テーブルデータを記述する必要がある, **When** ルールを参照する, **Then** パイプテーブルではなくlist-tableディレクティブで記述する
2. **Given** 外部ファイルに存在するコードを取り込む必要がある, **When** ルールを参照する, **Then** インラインコードブロックではなくliteralincludeで取り込む
3. **Given** 太字見出し＋箇条書きが複数並ぶパターンがある, **When** ルールを適用する, **Then** list-tableに変換する

---

### User Story 3 - Admonition（注意喚起ボックス）を活用する (Priority: P3)

コンテンツ作成者がスライドやドキュメント内で重要な情報、注意点、補足情報を強調する際、MyST Admonitionディレクティブ（note, warning, tip, importantなど）を使って視覚的に区別されたボックスとして表現する。

**Why this priority**: スライド内での情報の視認性と構造化に寄与し、重要な注意点や補足をプレーンテキストから視覚的に区別できる。

**Independent Test**: Admonitionディレクティブを含むスライドをビルドし、スタイル付きボックスとして表示されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 重要な補足情報を記述する必要がある, **When** ルールを参照する, **Then** 適切なAdmonitionタイプを選択して記述できる
2. **Given** 「注意」「ヒント」などの情報がプレーンテキストで埋もれている, **When** ルールを適用する, **Then** Admonitionボックスとして視覚的に強調される

---

### User Story 4 - 定義リストでキーワードを構造化する (Priority: P3)

コンテンツ作成者が用語や概念の説明を記述する際、定義リスト（deflist）構文を使って「用語: 説明」の形式で構造化する。

**Why this priority**: 用語説明は箇条書きより定義リストが意味的に正確であり、HTMLの`<dl>`要素として構造化される。

**Independent Test**: 定義リスト構文で記述したドキュメントをビルドし、HTML上で定義リストとしてレンダリングされることを確認できる。

**Acceptance Scenarios**:

1. **Given** 用語とその説明を列挙する必要がある, **When** ルールを参照する, **Then** 定義リスト構文で記述できる
2. **Given** 定義リストが箇条書きで代用されている, **When** ルールを適用する, **Then** 定義リストに変換される

---

### Edge Cases

- Mermaidでサポートされていないダイアグラム種別を記述しようとした場合、ビルドエラーにならず適切な代替手段が示されているか
- Mermaidダイアグラム内の日本語テキストが正しくレンダリングされるか
- list-table内にMermaidやAdmonitionをネストする必要がある場合の記法
- literalincludeのパスが存在しない場合のエラー検知

## Requirements *(mandatory)*

### Functional Requirements

#### Mermaidダイアグラム

- **FR-001**: フロー、シーケンス、状態遷移などの図表は `{mermaid}` ディレクティブで記述しなければならない
- **FR-002**: ASCIIアートやテキストベースのフローチャート（罫線文字による図）は使用禁止とする
- **FR-003**: Mermaidダイアグラムの推奨タイプを定義する（flowchart, sequenceDiagram, stateDiagram-v2, gantt, classDiagram）。推奨タイプ以外（pie, gitGraph, mindmap等）も使用を禁止しない
- **FR-004**: 本文から相互参照するダイアグラムにはラベル（`name`オプション）を付与する。相互参照が不要な場合はラベルを省略してよい

#### テーブル

- **FR-005**: テーブルは `{list-table}` ディレクティブで記述しなければならない（パイプテーブル禁止）
- **FR-006**: `:header-rows:` オプションを必ず指定する
- **FR-007**: `:widths: auto` を基本とする
- **FR-008**: 太字見出し＋箇条書きの複数グループは `{list-table}` に変換する
- **FR-008a**: テーブルの列数は3列以内とする。4列以上になる場合はスライドを分割する
- **FR-008b**: セル内の文字数が多くテーブルでは視認性が悪い場合もスライドの分割を検討する
- **FR-008c**: 列を横断して1対1で対応する比較項目は、セル内箇条書きではなく独立したテーブル行として記述する

#### コード取り込み

- **FR-009**: 外部ファイルに存在するコードは `{literalinclude}` ディレクティブで取り込む。ただし、説明用の5行以下のコード片はインラインコードブロックを許容する
- **FR-010**: `:language:` オプションを必ず指定する

#### Admonition

- **FR-011**: 強調すべき情報（注意、ヒント、重要事項）は Admonitionディレクティブで記述する
- **FR-012**: 使用可能なAdmonitionタイプを定義する（note, tip, warning, important, seealso）
- **FR-013**: Admonition内のコンテンツは1〜3行程度に簡潔にまとめる

#### 定義リスト

- **FR-014**: 「用語: 説明」形式の列挙は定義リスト構文で記述する
- **FR-015**: 単純な箇条書きで定義リストを代用しない

#### 一般ルール

- **FR-016**: ディレクティブの囲みにはコロンフェンス（`:::`）を使用する（バッククォートフェンスではなく）
- **FR-017**: ルール文書内に正しい記法と禁止パターンの両方を明示する

### Key Entities

- **ルール文書（SKILL.md）**: MyST記法のルールを定義するドキュメント。各ルールカテゴリ（テーブル、コード、ダイアグラム、Admonition、定義リスト）ごとにセクションを持つ
- **ディレクティブ**: MyST記法における構造化コンテンツの基本単位。`:::{directive-name}` 構文で囲む
- **禁止パターン**: ルールに違反する記法の具体例。ルール文書内で明示的に示す
- **正しい記法**: ルールに準拠した記法の具体例。禁止パターンと対で示す

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ルール文書がすべてのルールカテゴリ（テーブル、コード取り込み、ダイアグラム、Admonition、定義リスト、一般）をカバーしている
- **SC-002**: 各ルールに「正しい記法」と「禁止パターン」の両方の例が含まれている
- **SC-003**: ルール文書に従って記述されたMyST文書がビルドエラーなしで生成される
- **SC-004**: ASCIIアートで記述されたフローチャートがルールに従ってMermaidに変換可能であることが確認できる
- **SC-005**: コンテンツ作成者（人間またはAI）がルール文書のみを参照して、正しいMyST記法でコンテンツを作成できる
